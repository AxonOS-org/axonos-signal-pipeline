# Limitations (v0.1.0)

Read this before drawing any conclusion from this repository.

## The numeric pipeline does not exist yet

v0.1.0 is a **type contract and conformance surface**, not a working signal
processor. Specifically:

- There is **no** DSP. No DC removal, bandpass, or notch filtering is
  implemented (roadmap v0.2.0).
- There is **no** feature extraction. `FeatureVector` is a placeholder type
  holding `f32` values; the deterministic fixed-point path is v0.3.0.
- There is **no** classifier. `ClassifierDecision` is a typed container; no
  model produces it in this repository (roadmap v0.4.0).
- There is **no** calibration (Euclidean Alignment, ZeroCalib) — v0.5.0.

## Scope boundaries

- The contract stops at `ClassifierDecision`. Conversion to the canonical
  `IntentObservation` (RFC-0006 §4) and its consent gating live in
  `axonos-kernel` / `axonos-consent`, not here.
- Nothing below the acquisition frame is modelled: no analog front-end, no ADC
  driver, no transport. `axon-bci-gateway` is the acquisition bridge.

## Data

- All fixtures are **synthetic**, generated by a documented LCG. They are
  useful for determinism, not for any statement about real neural signals.
- No public dataset is vendored. Dataset compatibility, when added, will be a
  separate, license-audited concern.

## Numerics

- `FeatureVector` uses `f32` at v0.1.0; floating point is **not** part of any
  deterministic conformance claim yet. v0.1.0 vectors checksum only integer
  sample data. A fixed-point path is required before features become
  conformance-pinned.

## Privacy boundary — what it does and does not guarantee

The sealed `BoundarySafe` trait and the redacted `Debug` enforce, at compile
time and in logs, that raw signal types do not cross the boundary *through this
crate's API*. They do **not** prevent a caller from copying raw samples out
through their own code, nor do they constitute consent enforcement — that is
`axonos-consent`'s role. See [`PRIVACY_BOUNDARY.md`](PRIVACY_BOUNDARY.md).
